### 二、V3.4 迁移计划（分阶段执行）

#### Phase 1: 订单与服务解绑 + 两阶段确认（最优先）

**目标**：
- 实现 `ORDER_PENDING_CONFIRMATION` 状态
- 实现两阶段确认机制（MODE 写入 → 1500ms 内确认运行）
- 修正 `ORDER_COMPLETED` 含义：仅表示"服务已启动确认"

**需要修改的文件**：
1. `app/src/main/java/com/carwash/carpayment/data/order/OrderState.kt`（如不存在则新建）
   - 新增 `ORDER_PENDING_CONFIRMATION` 枚举值
   - 修正 `ORDER_COMPLETED` 注释

2. `app/src/main/java/com/carwash/carpayment/data/config/TimeoutConfig.kt`（新建）
   - 创建超时配置单例
   - 定义 `PLC_CONFIRM_TIMEOUT_MS = 1500L`（默认值）

3. `app/src/main/java/com/carwash/carpayment/ui/viewmodel/WashFlowViewModel.kt`
   - 修改 `Start_SendingMode` 状态处理：MODE 写入成功后立即设置 `ORDER_PENDING_CONFIRMATION`
   - 实现 1500ms 内检测 214=1 或 102≠0 的逻辑
   - 超时未确认 → `ORDER_MANUAL` + `ServiceTicket` + `failureReasonCode=PLC_CONFIRM_TIMEOUT`

4. `app/src/main/java/com/carwash/carpayment/ui/viewmodel/PaymentViewModel.kt`
   - 修正 `ORDER_COMPLETED` 相关注释和逻辑
   - 移除"订单完成依赖服务结束"的逻辑

**关键逻辑说明**：
- MODE 写入成功 → `OrderState = ORDER_PENDING_CONFIRMATION`
- 1500ms 内检测到 214=1 或 102≠0 → `OrderState = ORDER_COMPLETED`
- 超时未检测到 → `OrderState = ORDER_MANUAL` + 生成 `ServiceTicket`

**验收步骤**：
1. 发送 MODE 后，logcat 应显示：`ORDER_STATE_CHANGE: ORDER_PAID -> ORDER_PENDING_CONFIRMATION`
2. 1500ms 内检测到运行态，logcat 应显示：`ORDER_STATE_CHANGE: ORDER_PENDING_CONFIRMATION -> ORDER_COMPLETED`
3. 超时未检测到，logcat 应显示：`ORDER_STATE_CHANGE: ORDER_PENDING_CONFIRMATION -> ORDER_MANUAL, reason=PLC_CONFIRM_TIMEOUT`

**风险点**：
- 需要确保 MODE 写入接口返回成功后才设置 `ORDER_PENDING_CONFIRMATION`
- 1500ms 超时检测需要精确计时，避免误判
- 需要确保 `ServiceTicket` 数据结构已定义

#### Phase 2: 现金统计切换为事件驱动（移除 GetAllLevels 依赖）

**目标**：
- 移除所有 GetAllLevels/baseline 差值用于收款统计的调用链
- 实现事件驱动累计：从 `CashEventResponse` 累加 `paidCents`
- 保持分层：设备事件 → 状态机 → 订单层写入

**需要修改的文件**：
1. `app/src/main/java/com/carwash/carpayment/data/cashdevice/CashAmountTracker.kt`
   - 重构为事件驱动：移除 `updateFromLevels()` 方法
   - 新增 `handleCashEvent(event: CashEventResponse)` 方法
   - 维护 `sessionPaidCents` 累加器

2. `app/src/main/java/com/carwash/carpayment/ui/viewmodel/PaymentViewModel.kt`
   - 移除 `billBaselineCents`、`coinBaselineCents` 相关代码
   - 移除 `baselineStore` 相关调用
   - 移除 `calcSessionDeltaCents()` 和 `calculateLevelsDeltaCents()` 调用
   - 实现事件订阅：从 `getDeviceStatus()` 返回的事件列表中提取 `CashEventResponse`
   - 调用 `CashAmountTracker.handleCashEvent()` 累加金额

3. `app/src/main/java/com/carwash/carpayment/data/cashdevice/CashDeviceRepository.kt`
   - 保留 `getAllLevels()` 方法（仅用于找零评估，不用于收款统计）
   - 移除 baseline 相关方法（或标记为 Deprecated）

**关键逻辑说明**：
- 进入 `Paying_Cash` 状态时，启动事件轮询
- 每次 `getDeviceStatus()` 返回事件列表，解析 `CashEventResponse`
- 累加 `STORED`、`STACKED`、`VALUE_ADDED`、`COIN_CREDIT` 事件的 `value` 字段
- `paidCents` 完全来自事件累加，不再使用 baseline 差值

**验收步骤**：
1. 支付 10€，投 20€，找零 10€ 的完整流程
2. logcat 应显示每次事件累加：`CASH_EVENT: STORED, value=2000, sessionPaidCents=2000`
3. 达到目标金额后，logcat 应显示：`PAYMENT_COMPLETE: paidCents=2000, targetCents=1000, changeCents=1000`
4. 验证不再出现 `GetAllLevels` 用于收款统计的日志

**风险点**：
- 需要确保 `getDeviceStatus()` 返回的事件列表格式正确
- 需要处理事件去重（避免重复累加）
- 需要确保事件累加与 UI 显示同步

### 三、依赖关系与执行顺序

```
Phase 1 (订单解绑)
    ↓
Phase 2 (事件驱动) ← 可并行
    ↓
Phase 3 (打印) ← 依赖 Phase 1（PaymentAuthorized 状态）
    ↓
Phase 4 (Admin Console) ← 可独立进行
```

**建议执行顺序**：
1. Phase 1（最优先，影响订单生命周期）
2. Phase 2（核心功能，影响现金支付）
3. Phase 3（依赖 Phase 1）
4. Phase 4（独立模块，可最后完成）
